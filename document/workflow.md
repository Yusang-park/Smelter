# 🔄 Smelter Harness — Planning-State Workflow

> 이 문서는 각 단계를 유저의 관점에서 보여준다. spec, implementation 모두 동기화가 되어야한다.

---

## 실행 모델 요약

| 축                     | 값         | 의미                                       |
| --------------------- | --------- | ---------------------------------------- |
| 시작 명령 / Magic Keyword | `/tasker` | planning state 생성/보정 시작 (Step 1-3)       |
|                       | `/feat`   | 생성된 task를 기반으로 구현 작업 시작 (Step 4-11)      |
|                       | `/qa`     | 버그 수정 (Step 4-8 , 11) - 보다 간략화된 workflow |
* Command를 사용한 진입 혹은 Magic Keyword 훅을 사용해 실행 모델에 진입한다.
* 기본적으로 tasker 모드로 진입하며, 발화를 토대로 plan 및 task를 생성한다.
* `tasker`를 통해 구현 계획이 완료된 이후 구현에 따라 `feat`, `qa`로 분류하여 곧 바로 구현을 수행한다.
* tasker는 소규모(기존 기능에 대한 단순 수정), 중규모(기존 기능에 새로운 기능을 붙이거나 리팩토링), 대규모(완전히 새로운 기능 수립)을 구분한다.
* `/feat` `qa` 로 진입시 기존 태스크를 진행할 것인지, 새로운 태스크를 만들것인지 여부를 물어보고 경우에 따라 `tasker`를 수행한다.
* `feat`는 migration, refactoring, 기존 기능 구현 수정 등 또한 포함한다.
* `qa`는 기본적으로 버그 수정, UI 수정 등 기획과 판단보다 테스트가 더 중요한 경우 수행한다.

## Task Model

| 필드       | 값                                              |
| -------- | ---------------------------------------------- |
| `type`   | `qa` / `feature` / `need-review`               |
| `status` | `pending` / `in_progress` / `blocked` / `done` |

Task는 업무 의미를 표현해야 하며, `/feat`나 `/qa` 같은 시작 명령을 담지 않는다.

**출력 규칙**: 각 단계 진입 시 반드시 영어 헤더 출력
```
--- Step N: [Step Name] ---
Goal: <한 줄 목표>
```

## 용어 정의 (Terminology)

> 아래 용어만 사용한다. "칸반 카드", "카드 생성", "태스크 큐" 등의 혼용 표현은 사용하지 않는다.

| 용어             | 정의                                                                                                 | 파일/위치                                                      |
| -------------- | -------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **Task**       | 작업 단위 (기능 하나 / 이슈 하나 / 버그 하나). 워크플로우의 처리 단위.                                                       | `.smt/features/<slug>/task/<task_slug>.md`                 |
| **Task Queue** | 미완료 Task 목록. `- [ ]` 체크박스로 상태 관리                                                                   | `.smt/features/<slug>/task/<task_slug>.md` (Task 파일 내에 위치) |
| **Plan**       | Feature 내의 실행 계획. Tasks들의 체크박스 트리 형태. 검증 단계가 포함되어있다. QA와 같이 작은 문제의 경우, Plan은 하나의 Task만을 가질 수 도 있다. | `smt/<slug>/task/plan.md` 내 `## Plan`                      |
| **Session**    | Claude Code 실행 컨텍스트 단위. **Task 1개 = 독립 서브에이전트 1개 = 독립 컨텍스트.**                                      | —                                                          |
| **시작 명령**      | 작업을 시작하는 사용자 진입점 (`/tasker` / `/feat` / `/qa`).                                                    | Claude Code command                                        |

### 독립 컨텍스트 원칙 (Context Isolation)

> **Task 하나는 반드시 독립된 서브에이전트로 실행한다.**

```
/ralph
features/<slug>/task/plan.md`
  - [ ] Task A  →  서브에이전트 #1 (독립 컨텍스트)  →  완료
  - [ ] Task B  →  서브에이전트 #2 (독립 컨텍스트)  →  완료
  - [ ] Task C  →  서브에이전트 #3 (독립 컨텍스트)  →  완료
```

**이유**: 여러 Task를 하나의 세션에서 순차 처리하면 컨텍스트가 비대해진다.  
Task A의 코드·로그·에러가 Task B 처리 시 잡음이 된다.  
독립 서브에이전트는 각 Task를 신선한 컨텍스트에서 시작하므로 이 문제를 원천 차단한다.

### 스코프 테스트 원칙 (Scoped Testing)

> **변경된 파일과 관련된 테스트만 실행한다. 전체 테스트 스위트는 실행하지 않는다.**

```bash
# 변경 파일 식별
git diff --name-only

# 단위/통합 테스트: 관련 파일만
npm test -- --testPathPattern="auth|profile"   # Jest 예시
pytest tests/test_auth.py tests/test_profile.py  # pytest 예시

# E2E: 관련 spec만
npx playwright test tests/auth.spec.ts tests/profile.spec.ts

# ❌ 금지 (명시적 요청 없이)
npm test          # 전체 단위 테스트
npx playwright test  # 전체 E2E
```

> 전체 회귀 테스트는 Step 11(Human Review) 완료 결정 시 또는 사용자가 명시적으로 요청할 때만 실행한다.

# Smelter — Core Workflow

> **정의**: 실제 인간 개발자의 워크플로우를 그대로 자동화한다.
> PM/QA 업무 수령 → 학습 → 계획 → TDD → 구현 → 검증 → 리뷰 → 사용자 확인 → 배포

## 필수 출력 규칙 (REQUIRED OUTPUT FORMAT)

각 Step 진입 시 반드시 다음 형식으로 영어 헤더를 출력한다:

```
--- Step 1: Problem Recognition ---
--- Step 2: Pre Review ---
--- Step 3: Planning ---
--- Step 4: Test Design (TDD) ---
--- Step 5: Implementation ---
--- Step 6: Agent Review ---
--- Step 7: Dry Test (Build + Unit) ---
--- Step 8: E2E Validation ---
--- Step 9: Team Code Review ---
--- Step 10: Human Review ---
```

**규칙:**
- 출력은 영어로만
- 단계 전환 시 반드시 헤더 출력 (건너뛰는 단계 제외)
- 헤더 뒤에 해당 단계의 간략한 목표 1줄 출력

## 워크플로우 상세 (10단계)

---
## Step 1 — Problem Recognition (문제 인식)

> **유사 역할**: QA/PM으로부터 업무를 수령하는 것

### 목적
문제를 인식하고 초안을 작성한다. 작업을 바로 수행하는 것이 아닌 기록으로 시작한다.

### 수행 절차
- 문제 혹은 기능 요청을 확인하고 Feature 디렉토리 생성
- `features/{slug}/task/plan.md` 초기화 (목표, 범위, acceptance criteria)
- `features/{slug}/task/{task-name}.md` 생성 (제목, 배경, 목표, 상태: `pending`)

**템플릿**
[[plan.md]]
[[task.md]]

**산출물**: 
* `plan.md` + Task 문서 (초안), 

### 시작 명령 적용
- `tasker`: 핵심 단계
- `feat`, `qa` : 요청한 작업에 대한 문서가 없으면 tasker 명령으로 먼저 진입한다.

---

## Step 2 — Pre Review (기획 검토)

> **유사 역할**: 엔지니어가 계획의 모호함, 모순이 없는지 검토한다. 구현에 필요한 솔루션 후보를 탐색한다.

### 목적
1. 구현 전에 접근 방식을 먼저 탐색한다. "일단 코딩부터"가 아닌 "먼저 조사 후 설계"의 원칙.
2. 수립된 기능에 대한 모호함이 없는지, 모순이 없는지 검토한다.
3. 기존 기능과의 충돌에 대한 리스크를 검토한다. 중복, 데이터베이스 레벨 등
4. 해당 단계에서 발견된 리스크를 모두 해결하고 넘어간다.

### 수행 절차
1. 문제 해결 가능한 접근 방식 2~4개 도출
2. 각 후보에 대해 기록:
   - 장단점
   - 구현 복잡도
   - 위험도
   - 적용 가능성
3. Task 문서 `## Approaches` 섹션에 작성
4. 최종 채택 방식 확정
5. 문서 수정

### 의사결정이 어려운 경우 — 95% 합의 프로세스
접근 방식에 대해 명확한 판단을 내리기 어려운 경우:

```
1. 서브 에이전트 3개 할당:
   - advocate  (긍정): 해당 방식의 장점을 최대한 옹호
   - critic    (부정): 해당 방식의 위험과 단점 지적
   - arbitrator(중립): 양측 의견 종합, 점수 계산

2. 각 라운드마다 동의율 계산
3. 모든 에이전트 동의율 ≥ 95%가 될 때까지 반복
4. 채택 방식 확정 → 문서에 근거와 함께 기록
```

> 이 프로세스는 인간 팀의 의견 수렴 방식과 동일하다.

### 산출물
- Task 문서 `## Approaches` 섹션 완성 및 기획 수정
- 채택 접근 방식 + 채택 이유

### 시작 명령 적용
- `tasker`: 포함 (핵심 단계)
- `feat`, `qa` : 해당 단계를 통과한 문서가 없으면 tasker 명령으로 먼저 진입한다.

---

## Step 3 — Planning (계획)

> **유사 역할**: 기술 설계 문서 작성

### 목적
실행 전에 계획을 먼저 완성한다. 계획 없이 코딩하면 방향이 흔들린다.

### 문서 구조
```markdown
## Queue
- [ ] 기능 A 구현
  - [ ] A-1: 데이터 모델 설계
  - [ ] A-2: API 엔드포인트 구현
  - [ ] A-3: UI 컴포넌트 구현
- [ ] 기능 B 구현 [parallel]
- [ ] TDD 작성 (최소 10개)
- [ ] E2E 작성

## Wiki Links
- [[기능A-상세설계]]
- [[API-명세]]

## Risks
(초기에는 비어있음 — Step 6, 9, 10에서 채워짐)

## Approaches
(Step 2에서 채워짐)
```

### 선형 구조 원칙
- 모든 기능은 **의존 관계 순서**대로 나열
- 병렬 처리 가능한 파트는 `[parallel]` 태그로 명시
- 이전 단계 완료 없이 다음 단계 진행 불가

### 병렬 처리 (독립 파트)
계획 내 독립적인 파트는 서브 에이전트로 분배:
```
[parallel] 파트 A → Sub-agent 1
[parallel] 파트 B → Sub-agent 2
동기화 포인트에서 합류
```
> 별도 `team` command가 아니라, **계획 내 병렬 처리** 방식이다.

### 산출물
- Task 문서 `## Queue` 완성 (체크박스 트리)
- Wiki 링크 연결

> 이 과정이 끝나면 유저와 추가 인터뷰를 진행합니다. 필요시 1-3단계를 오갈 수 있습니다.

### 시작 명령 적용
- `tasker`: 1,2단계 완료 후 `feat`, `qa` 로 전환하여 수행한다.
- `feat`, `qa` : 진행하고자 하는 feature를 지목하여 수행한다.

## Step 4 — Test Design / TDD (테스트 설계)

> **스킬**: `tdd-linear` 사용

### 목적
구현 전에 테스트를 먼저 설계한다. 테스트가 없으면 구현을 시작할 수 없다.

### 수행 절차
1. 구현할 기능마다 테스트 파일 생성 (구현 파일 없이)
2. 최소 10개 이상의 테스트 작성:

| 카테고리 | 최소 수 | 예시 |
|---------|---------|------|
| 정상 동작 (happy path) | 2+ | 올바른 입력으로 정상 결과 |
| 경계 조건 (boundary) | 2+ | 빈 값, 최소/최대값 |
| 실패 케이스 (error path) | 2+ | 잘못된 입력, 에러 처리 |
| 엣지 케이스 (edge case) | 2+ | 특수 문자, 동시성, 예외 |
| 통합 (integration) | 1+ | 다른 모듈과의 상호작용 |

3. 모든 테스트 실행 → 반드시 **FAIL** 상태 확인 (RED)

### TDD 면제 조건

| 변경 유형 | TDD 필요? |
|---------|---------|
| CSS/스타일/색상/타이포그래피 | ❌ 불필요 |
| i18n/번역 텍스트 변경 | ❌ 불필요 |
| 기존 기능 수정 | ✅ 필요 (기존 테스트 먼저 확인) |
| 신규 기능 구현 | ✅ 필요 |

### 기존 기능 수정 시 절차
1. 관련 기존 테스트 검색
2. 수정으로 인해 실패해야 할 테스트 확인
3. 해당 테스트를 **먼저 실패하도록 수정** (RED)
4. 그 다음 구현 진행

### 산출물
- 테스트 파일들 (모두 RED 상태, 구현 파일 없음)

### 시작 명령 적용
Step 4와 동일

---

## Step 5 — Implementation (구현)

> **유사 역할**: 실제 코딩

### 목적
Step 3 계획 + Step 4 TDD를 기반으로 코드를 작성한다. 계획과 테스트 없이 이 단계에 도달할 수 없다.

### 수행 절차
1. 테스트를 통과시키는 **최소 구현** (GREEN)
2. 리팩토링 (REFACTOR) — 테스트 통과 유지하며 코드 정리
3. Task 문서 `## Plan`에서 구현 완료 항목 체크:
   ```
   - [x] A-1: 데이터 모델 설계  ← 체크
   - [ ] A-1: 검증 통과         ← 검증 전까지 체크 안 함
   ```

### 실패 시 재진입 규칙

| 상황 | 조치 |
|------|------|
| 검증 실패 후 재진입 | 반드시 Task 문서 업데이트 후 진행 |
| 동일 접근 방식 3회 이상 실패 (`work`) | **Step 2로 복귀** — 다른 방식 채택 |
| 동일 접근 방식 3회 이상 실패 (`default`) | **`work` 수준으로 에스컬레이션** — 더 넓은 조사와 검증 적용 |
| "하는 김에..." 추가 구현 | **금지** — 계획에 없는 것은 Step 3로 돌아가 추가 |

### 산출물
- 구현 코드 (테스트 통과)
- Task 문서 구현 완료 체크박스 업데이트

### 시작 명령 적용
Step 4와 동일

---

## Step 6 — Local Agent Review (로컬 에이전트 리뷰)

> **유사 역할**: 코드 완성 후 셀프 리뷰

### 목적
구현 직후 서브 에이전트들이 독립적으로 검토한다. 사람이 자신의 코드를 바로 검토하면 놓치는 것이 있듯, 별도 에이전트가 검토한다.

**이 과정은 반드시 3번 반복한다. 3번 연속으로 통과하지 못하지 못하고 이전 Step으로 복귀하는 경우 다시 3번을 수행한다.**

### 검토 항목 및 복귀 규칙

| 검토 항목                     | 실패 시 복귀 |
| ------------------------- | ------- |
| 코드 품질 (가독성, 네이밍, 복잡도)     | Step 5  |
| 버그 및 로직 오류                | Step 5  |
| 보안 취약점                    | Step 5  |
| 계획(`## Plan`) 대비 구현 누락    | Step 3  |
| 엣지 케이스 (보안, QA 이슈, 실패 처리) | Step 3  |

### Risk 기록
모든 발견 사항은 **Task 문서 `## Risks` 섹션**에 기록:
```markdown
## Risks
- [LOW] 에러 메시지가 사용자에게 너무 기술적으로 노출됨
- [MEDIUM] 동시 요청 시 race condition 가능성
- [HIGH] 인증 토큰 만료 처리 누락
```

### 산출물
- `## Risks` 업데이트
- 이슈 없으면 Step 7로 진행

## Step 7 — Utility Test

> **유사 역할**: CI/CD 파이프라인의 자동 검증

### 목적
코드가 올바르게 작동하는지 환경 레벨에서 검증한다. Test code, build, typescript 체크와 같은 최소 수준의 코드 검증이다.

### 검사 항목 체크리스트

> ⚠️ **스코프 제한 필수** — 변경된 파일과 관련된 테스트만 실행한다. ([스코프 테스트 원칙] 참조)

| 항목                 | 명령 예시                                                                         | 실패 시      |
| ------------------ | ----------------------------------------------------------------------------- | --------- |
| 유닛/통합 테스트 (관련 파일만) | `npm test -- --testPathPattern="<keyword>"` / `pytest tests/test_<module>.py` | Step 5 복귀 |
| 빌드 성공              | `npm run build`                                                               | Step 5 복귀 |
| 타입 검사 (TypeScript) | `tsc --noEmit`                                                                | Step 5 복귀 |


### 산출물
- 테스트/빌드 결과 로그
- 모두 통과 시 Step 8로 진행

### 시작 명령 적용
Step 4와 동일

## Step 8 — E2E Validation (E2E 검증)

> **유사 역할**: 개발자의 QA 과정

### 목적
실제 사용자 시나리오로 검증한다. 단순 유닛 테스트로는 잡지 못하는 통합 문제를 발견한다.
E2E는 Interface test로 정의된다. 즉, Playwrite와 같은 테스트 뿐만 아니라 API 테스트, 기능 실제 검증 테스트와 같은 실제 구현이 작동하는지에 대한 평가를 진행해야한다.

### E2E 실행 방식 — 5-Surface Mapping

| Component      | Real Interface           | Runner                     |                |
| -------------- | ------------------------ | -------------------------- | -------------- |
| UI/Frontend    | Browser                  | Playwright                 |                |
| CLI/Script     | stdin, argv, exit code   | subprocess                 |                |
| HTTP API       | HTTP endpoints           | real server + curl/fetch   |                |
| Database/Query | Real DB queries          | real or in-process test DB |                |
| Hook script    | stdin JSON → stdout JSON | `cat payload               | node hook.mjs` |

상세:
1. **직접 API 호출 검증** — REST/GraphQL 엔드포인트
2. **Playwright UI 검증** — 브라우저 시나리오
3. 이하 직접 도구 사용


### 제약 사항 (중요)
| 제약 | 상세 |
|------|------|
| PROD 데이터 수정/삭제 금지 | 사용자 허락 없이 절대 불가 |
| 테스트 데이터셋 확인 | test용 데이터셋 확인 후 자동 허용 |
| 로그인 필요 시 | `.env`에 E2E auth 정보 삽입 요청 |
| 데이터 없을 시 | 사용자에게 데이터 삽입 요청 후 대기 |

### 산출물 형태

| 타입 | 산출물 | 필수 여부 |
|------|--------|---------|
| 프론트엔드 | 비디오 녹화 + 스크린샷 | **필수** |
| 백엔드/API | 로그 파일 | **필수** |

저장 경로: `.linear-harness/features/<slug>/artifacts/`

### 실패 시
**Step 5로 복귀** — E2E 실패 내용을 Task 문서 `## Risks`에 기록

### 시작 명령 적용
Step 4와 동일

## Step 9 — Team Code Review (팀 에이전트 코드 리뷰)

> **유사 역할**: 실제 팀 코드 리뷰

### 목적
3개의 독립된 관점에서 전체 구현을 재검토한다. 단일 관점의 리뷰는 편향이 생기므로, 긍정/부정/중립 역할로 분리한다. Step 8에서 완료된 Video, Log, Screenshot 등과 함께 다시 한번 리뷰한다.

### 3-에이전트 합의 프로세스

```
에이전트 역할:
  advocate  (긍정): 구현의 장점, 올바른 결정, 잘된 점 옹호
  critic    (부정): 문제점, 놓친 케이스, 잠재적 버그 지적
  arbitrator(중립): 양측 의견 종합, 점수 계산, 최종 판단

라운드 반복 조건:
  → 모든 에이전트 동의율 ≥ 95% 달성까지 계속
  → 각 라운드마다 새로운 관점 제시 가능
```

### 검토 범위
1. Step 6 리뷰 사항 **재검토** (해소 여부 포함)
2. **새로운 문제점** 탐색
3. 보안 취약점, 성능 이슈, 유지보수성 평가
4. 기획에 대한 구현이 올바른지 검토

### 심각도 분류 및 복귀 규칙

| 심각도        | 기준                        | 조치                         |
| ---------- | ------------------------- | -------------------------- |
| `CRITICAL` | 데이터 손실, 보안 취약점, 서비스 중단 가능 | **Step 3 복귀** (재설계)        |
| `HIGH`     | 주요 버그, 심각한 엣지 케이스 누락      | **Step 3 복귀**              |
| `MEDIUM`   | 개선 필요한 로직, 마이너 버그         | **Step 5 복귀** (수정)         |
| `LOW`      | 스타일, 네이밍, 제안 사항           | Task 문서 `## Risks` 기록 후 계속 |

### 산출물
- 팀 리뷰 결과 (심각도 분류 포함)
- 이슈 없으면 Step 11로 진행

### 시작 명령 적용
* `qa`의 경우 건너뜀

## Step 10 — Human Review (사용자 리뷰)

> **유사 역할**: 프로젝트 매니저·작성자 최종 확인 (실제 "writer"의 역할)

### 목적
모든 자동화된 검증이 끝난 후, 사람이 최종 확인한다. AI가 놓칠 수 있는 비즈니스 맥락, 사용성, 의도 부합 여부를 사람이 판단한다.

### 10-1. 검증 자료 제시

사용자에게 다음 보고서 형식으로 제시:
[[report.md]]

### 10-2. 사용자 결정

```
선택지:
  [1] 재작업  → 재작업 범위를 명시 → Step 3으로 복귀
  [2] 완료    → Git 옵션으로 이동
  [3] 보류    → features/<slug>/task.md에 `blocked` 또는 보류 사유 기록, 나중에 재개
```

**재작업 선택 시**: 사용자가 재작업 이유와 범위를 구체적으로 명시하면 Task 문서에 기록 후 Step 3로 복귀.

### 10-3. Git 옵션 (완료 선택 시)

```
Git 옵션:
  [a] 현재 브랜치에 push
        → git push origin {현재 브랜치}

  [b] 새 브랜치 생성 후 push
        → 브랜치명 입력
        → git checkout -b {name} && git push -u origin {name}

  [c] 로컬 완료 (push 없음)
        → 커밋만 하고 push하지 않음
```

선택 후 결과(커밋 해시 / PR URL) 보고.

### 10-4. Task 문서 마무리

완료 확정 시:

```
1. features/<slug>/task/plan.md 업데이트:
   - 모든 Plan 체크박스 확인 완료
   - status: done 기록
2. features/<slug>/results.md 기입:
     Video: {경로}
     Log:   {경로}
     Branch: {브랜치명}
     PR:    {URL 또는 "없음"}

3. features/<slug>/task/plan.md 업데이트:
   - [ ] Task {id}: {제목}
     →
   - [x] Task {id}: {제목}

3. 세션 로그 기록:
   - .smt/sessions/{YYYY-MM-DD}.md 에 추가:
     ## {HH:MM} Task {id} 완료 — {제목}
```
